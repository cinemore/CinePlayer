## Context

当前仓库的 macOS 控制面板仍是早期纯播放器布局：顶部只含关闭与部分播放器按钮，底部播放控制保留“上一集/下一集”，未对齐 iOS 当前已落地的业务按钮集（字幕、设置、增强、信息等入口）。同时，`SiderView` 与 `PlayerControlModel` 已具备所需容器状态，macOS 主要缺口在控制面板入口编排和交互触发。

## Goals / Non-Goals

**Goals:**
- macOS 控制面板功能与 iOS 对齐：关闭、PiP、画面填充、快退/快进、播放/暂停、音轨、视频轨、倍速、字幕、设置、增强、媒体信息。
- 保持 macOS 交互稳定：保留全屏切换，不引入 iOS 横竖屏切换按钮。
- 面板按钮与侧栏状态联动统一使用 `PlayerControlModel`，避免局部状态分裂。

**Non-Goals:**
- 不引入换源、剧集列表等业务模型。
- 不改动 iOS/tvOS/visionOS 的既有布局策略。
- 不新增外部依赖或跨仓库 API。

## Decisions

### Decision 1: 以 iOS 面板分组语义重排 macOS 顶/底栏
- 选择：将 macOS 分为顶部（关闭 + PiP/填充/全屏 + 信息/增强/设置）和底部（进度条 + 播放控制 + 倍速/字幕/音轨/视频轨）两层。
- 原因：直接对齐 iOS 已验收的业务分组，降低跨端学习成本。
- 备选：在现有 macOS 布局上增量补按钮。该方式会继续保留“上一集/下一集”语义，与目标冲突。

### Decision 2: 保留 macOS 特有全屏按钮，但不替代 PiP/填充
- 选择：PiP 与画面填充按钮始终可见（按能力可用性显示），全屏作为 macOS 特有补充按钮保留。
- 原因：满足“除横竖屏外其他对齐”的同时，不丢失桌面平台核心能力。
- 备选：全屏与 PiP/填充互斥。会造成与 iOS 能力集合不一致。

### Decision 3: 统一底部播放控制为快退/播放暂停/快进
- 选择：移除 macOS 面板里的上一/下一集按钮，改为基于 `PlayerControlConfig` 的秒级快退/快进。
- 原因：与 iOS 一致，并符合用户明确要求。
- 备选：保留上一/下一集并额外增加快退/快进。会导致控件拥挤且语义混杂。

## Risks / Trade-offs

- [风险] macOS 按钮数量增加后小窗口布局可能拥挤。  
  -> Mitigation：保留现有容器宽度阈值策略，在窄宽度下保持播放组与业务组分离。

- [风险] 新增按钮联动更多容器，掩层显隐节奏可能变复杂。  
  -> Mitigation：所有入口先调用 `hideContainer()` 再激活目标容器，保持单容器打开。

- [风险] 全屏切换时播放器暂停/恢复时机抖动。  
  -> Mitigation：沿用现有 `toggleFullScreen()` 的暂停与延迟恢复策略。

## Migration Plan

1. 更新 `ControllerPanelViewMacOS` 的按钮分组和触发逻辑。
2. 替换底部播放组为快退/播放暂停/快进实现。
3. 对照检查 Sider 容器入口映射（字幕/设置/增强/音轨/视频轨/倍速）。
4. 执行 7 平台 `xcodebuild` 验证并记录结果。

回滚：仅回退 `ControllerPanelViewMacOS.swift` 相关修改即可恢复旧行为。
